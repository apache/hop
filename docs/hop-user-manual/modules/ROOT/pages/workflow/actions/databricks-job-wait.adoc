= Databricks Job Wait

== Description

**Databricks Job Wait** polls the Jobs API (`jobs/runs/get`) until a run reaches a terminal state (`SUCCESS`, `FAILED`, тАж). Use it after xref:workflow/actions/databricks-job-run.adoc[Databricks Job Run] in **Fire and forget** mode, or when another system already started a run and you only have a **Run ID**.

There is no dedicated тАЬwaitтАЭ endpoint on Databricks: this action repeatedly GETs run status, sleeps for the poll interval, and exits when the run finishes or the timeout is hit.

== Typical pattern

. Job Run with **Fire and forget** тЖТ sets `${DatabricksRunId}`
. Other workflow actions (notifications, parallel prep, тАж)
. **Job Wait** with Run ID `${DatabricksRunId}` тЖТ success/failure hops

For тАЬsubmit and block on this step,тАЭ use Job RunтАЩs built-in **Wait for completion** instead.

== Options

[options="header",cols="1,3"]
|===
|Option |Description

|Databricks connection
|xref:metadata-types/databricks-connection.adoc[Databricks Connection] (PAT)

|Run ID
|Numeric run id; default `${DatabricksRunId}`

|Timeout (seconds)
|Maximum wait; `0` = no timeout

|Poll interval (seconds)
|Delay between status checks (default 15)

|Cancel run if workflow is stopped
|If checked, calls `jobs/runs/cancel` when the parent workflow stops

|Result variables
|Same defaults as Job Run: `DatabricksJobId`, `DatabricksRunId`, `DatabricksStatus`, `DatabricksRunPageUrl`, `DatabricksError`
|===
